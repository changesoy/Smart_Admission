## 二、角色与责任划分

### 角色 A · 地图与查询工程师

**核心使命**:让"找到自己家学区"这件事变得快速、准确、可追溯。

**负责功能**
1. 学区地图扩展(从 5 个示例 → 10-15 个,使用真实道路名称参考)
2. 关键词检索(输入小区/路名 → 自动匹配学区 + 地图飞到该学区)
3. 图层切换控件(矢量底图 ↔ 影像底图)
4. 学区历年调整记录(在结果面板加 timeline)

**专属数据文件**
- `data/zones.geojson`(扩展)
- `data/zones_history.json`(新增)
- `data/keywords_index.json`(新增,小区/路名 → zoneId 映射)

**专属 JS 文件**
- `js/mapService.js`(已有,需扩展)
- `js/searchService.js`(新增)

---

### 角色 B · 政策与模拟工程师

**核心使命**:让家长理解政策变化、预判录取结果、不被谣言带偏。

**负责功能**
1. 历史政策归档(扩展到 2022-2025 共 4 年)
2. 政策对比视图(选两个年份,并排展示差异点)
3. 录取建议模拟器(输入户籍/房产/居住年限 → 输出推荐学区与录取概率参考)
4. 政策辟谣模块(谣言 + 官方澄清卡片)

**专属数据文件**
- `data/policies.json`(扩展)
- `data/policy_diff.json`(新增)
- `data/rumors.json`(新增)
- `data/simulator_rules.json`(新增)

**专属 JS 文件**
- `js/policyService.js`(新增,从 render.js 拆出政策部分)
- `js/simulatorService.js`(新增)

---

### 角色 C · 互动与内容工程师

**核心使命**:让家长能完成"咨询 + 准备材料 + 解疑"的全流程。

**负责功能**
1. 政民互动模块(留言表单 + 教育局联系卡片)
2. 入学材料清单增强(模板下载链接 + 常见被拒原因 + 有效期标签)
3. FAQ 增强(分类筛选 chip + 相关问题推荐)
4. 整体 UI 体检(响应式微调 + 加载骨架屏 + 错误状态美化)

**专属数据文件**
- `data/materials.json`(扩展)
- `data/faq.json`(扩展)
- `data/contacts.json`(新增)

**专属 JS 文件**
- `js/materialService.js`(新增,从 render.js 拆出材料部分)
- `js/faqService.js`(新增,从 render.js 拆出 FAQ 部分)
- `js/interactionService.js`(新增)

## 四、共享文件的并行策略

### 4.1 `index.html` 用注释分区(强制约定)

```html
<body>
  <!-- 导航条与标题区(共享,谁都可以加 badge) -->
  ...

  <!-- ========== A: 地图查询区 BEGIN ========== -->
  <section id="section-map">...</section>
  <section id="section-search">...</section>
  <section id="section-zone-history">...</section>
  <!-- ========== A: 地图查询区 END ========== -->

  <!-- ========== B: 政策模拟区 BEGIN ========== -->
  <section id="section-policy">...</section>
  <section id="section-policy-diff">...</section>
  <section id="section-simulator">...</section>
  <section id="section-rumors">...</section>
  <!-- ========== B: 政策模拟区 END ========== -->

  <!-- ========== C: 互动内容区 BEGIN ========== -->
  <section id="section-materials">...</section>
  <section id="section-faq">...</section>
  <section id="section-interaction">...</section>
  <!-- ========== C: 互动内容区 END ========== -->

  <!-- 页脚 + script 引用(共享) -->
</body>
```

**规则**:
- 每人只改自己 BEGIN/END 区块内的内容
- 区块外的改动(脚本引用、页脚、Bootstrap class)必须发到群里通知一声
- 合并冲突几乎只会出现在 script 引用顺序处,由组长统一管理

### 4.2 `css/style.css` 用类名前缀分区

| 前缀 | 归属 | 示例 |
|---|---|---|
| `.map-*` `.zone-*` `.search-*` | A | `.map-toolbar`、`.zone-history-timeline` |
| `.policy-*` `.sim-*` `.rumor-*` | B | `.policy-diff-row`、`.sim-result-card` |
| `.mat-*` `.faq-*` `.contact-*` | C | `.mat-template-btn`、`.faq-chip-active` |
| `:root` 变量、`.section-card`、`.section-header` | 共享(组长改) | `--primary-color` 等 |

### 4.3 `js/main.js` 协调入口

```javascript
document.addEventListener('DOMContentLoaded', function () {
  showLoading(true);
  DataService.loadAllData()
    .then(function (data) {
      showLoading(false);

      // 通用渲染
      RenderService.renderStats(data);
      RenderService.setBoundaryNotice();

      // ===== A 的模块 =====
      MapService.initMap({ ... });
      SearchService.init(data);

      // ===== B 的模块 =====
      PolicyService.init(data);
      SimulatorService.init(data);

      // ===== C 的模块 =====
      MaterialService.init(data);
      FaqService.init(data);
      InteractionService.init(data);
    })
    .catch(function (err) {
      RenderService.renderError(err.message);
    });
});
```

**规则**:每人的 service 必须暴露 `init(data)` 方法,接口签名一致,组长合并时不需要思考。

## 五、Git 工作流

### 5.1 分支策略

```
main
├── feature/A-map           (A 长期分支,A 在此推送)
├── feature/B-policy        (B 长期分支,B 在此推送)
├── feature/C-interaction   (C 长期分支,C 在此推送)
└── hotfix/*                (紧急修复用,谁都能开)
```

- `main` 保护:只能通过 PR 合并,禁止直推
- feature 分支:每天本地多次 commit,推送到 GitHub 至少一次
- 每周(M1/M2/M3 节点)合并到 main,合并前必须本地跑通

### 5.2 PR 流程

1. 在自己 feature 分支完成功能 → 推送到 GitHub
2. 在 GitHub 上发起 PR(feature/X → main),标题写 `[W2][A] 添加关键词检索功能`
3. 另两人至少 1 人 review,通过后合并
4. 合并后,另两人立刻 `git pull origin main` 同步到自己的 feature 分支
5. 解决合并冲突(通常只有共享文件)

### 5.3 Commit 信息规范

```
[W2][A] feat: 添加关键词检索输入框
[W2][B] fix: 修复政策对比视图年份排序错误
[W2][C] style: FAQ 卡片间距调整
[W3][A] refactor: mapService 拆分图层切换逻辑
```

**前缀含义**:
- `feat` 新功能
- `fix` Bug 修复
- `style` 仅样式调整
- `refactor` 重构(不改变行为)
- `docs` 文档
- `chore` 杂项

### 5.4 防冲突铁律

1. **每天开始工作前先 `git pull origin main`**,保持本地最新
2. **改共享文件前先在群里喊一声**(只需 30 秒),避免两人同时改 `main.js`
3. **每次推送前自测一遍**,不要把 broken state 推上去
4. **不要在 feature 分支待超过 3 天不推送**,憋大招最容易冲突

## 七、AI-IDE 辅助使用建议

三人都有 AI 辅助,但要避免几个常见坑:

1. **AI 容易"擅自重构"**:每次让 AI 改代码,明确告诉它"只改 X 函数,不要动其他地方",避免它顺手重写一堆
2. **AI 容易破坏锁定的版本号**:每次 AI 输出代码前,提醒它"必须使用 Leaflet 1.9.4 + Turf 6.5.0,不允许换"
3. **AI 容易引入 ES Module**:让它写 Vanilla JS,看到 `import/export` 立刻拒绝
4. **AI 容易"过度智能"**:它会很想加防抖、节流、缓存优化等,如果不需要就直接说"不用"
5. **PR review 时认真看 AI 生成的代码**:特别注意它有没有偷偷改你的字段名(`zoneId` 变成 `zone_id`)、有没有改坐标顺序

**推荐 prompt 模式**:
```
我在做"智慧入学·学区治理一站式可视化门户"项目,纯前端,
使用 Leaflet 1.9.4 + Turf 6.5.0 + Vanilla JS(无任何框架,无 import/export)。
当前文件:js/searchService.js
我需要添加 [具体功能]。
约束:
1. 字段名严格使用 zoneId / zoneName,不要改
2. 暴露 window.SearchService = { init }
3. 不要引入新的依赖
请只生成这个文件的代码。
```

## 八、关键风险与应对

| 风险 | 概率 | 影响 | 应对 |
|---|---|---|---|
| W1 schema 没定好,W2 返工 | 中 | 高 | W1 D1 严格闭门讨论 4 小时,所有字段拍板 |
| 三人风格不一致(配色/字体/间距) | 高 | 中 | W1 共同定 CSS 变量,严禁个人改 `:root` |
| 共享文件(index.html / main.js)冲突 | 中 | 中 | 注释分区 + 提前打招呼 + 频繁集成 |
| 天地图 token 失效或限流 | 低 | 高 | 备用方案:保留 OSM 切换接口,在 config 里可一键切换 |
| 录取模拟器规则过于复杂做不完 | 中 | 中 | 先做最简化版(3 个输入 → 1 个建议),够答辩展示即可 |

| 答辩前一晚发现 Bug | 高 | 高 | M3 后冻结新功能,W4 只允许 fix |
| 个别成员进度严重落后 | 中 | 高 | M2 节点强制 demo,落后者立刻重新分配任务 |
